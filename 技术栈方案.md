# 社区产品技术栈方案（V1.0）

## 1. 方案目标

本方案基于当前 PRD 和 User Story 的 V1 范围制定，目标是在两人团队前提下，兼顾以下四点：

1. 开发速度足够快，能优先完成注册、发帖、购买、悬赏、评论、通知、后台运营等核心闭环。
2. 金币交易、悬赏冻结与退回具备较强的数据一致性，避免重复扣费、重复发奖和资产异常。
3. 架构保持简单，便于早期快速迭代，不引入微服务、消息中间件集群等过重基础设施。
4. 为后续接入第三方支付、积分体系、推荐系统和更复杂的运营能力预留扩展空间。

## 2. 技术选型原则

### 2.1 核心原则

1. 单体优先
   V1 优先采用模块化单体架构，不拆微服务。原因是当前业务体量和团队规模都不需要分布式系统，过早拆分只会增加部署和协作复杂度。

2. 强事务优先
   金币、订单、悬赏、退款都属于强一致性场景，核心资产和交易数据统一落在 PostgreSQL 中管理，不将 Redis 作为最终数据源。

3. 异步解耦但不复杂化
   通知、过期检查、延迟处理等非主链路能力通过 Redis + 轻量任务队列处理，不引入 Kafka、RocketMQ 这类重型组件。

4. 前后端分层清晰
   前端负责渲染、交互与状态管理；后端负责业务规则、权限控制、资产事务、审核与后台管理。

5. 先满足 V1，后考虑扩展
   搜索、推荐、支付、埋点、风控先保留扩展位，当前只选 V1 够用且能持续演进的技术。

## 3. 推荐总体架构

### 3.1 架构形态

推荐采用以下整体架构：

- 前端：Next.js Web 应用，前台站点和后台管理共用一套前端工程。
- 后端：Spring Boot 模块化单体服务，按领域模块拆分代码。
- 主数据库：PostgreSQL。
- 缓存与基础中间件：Redis + Redisson。
- 异步任务与调度：Redis Stream + Quartz。
- 对象存储：本地开发用 MinIO，生产建议使用 S3 兼容对象存储。
- 反向代理：Nginx。
- 部署方式：Docker Compose 起步，后续可迁移到容器平台。

### 3.2 模块划分

后端按以下领域模块组织：

- auth：注册、登录、鉴权、Token 刷新
- user：用户资料、关注关系、账号状态
- wallet：钱包、金币流水、后台加币扣币
- post：普通帖、资源帖、悬赏帖
- trade：资源购买、申诉退款、交易记录
- bounty：悬赏冻结、采纳、超时退回
- comment：评论、回复、删除、@
- engagement：点赞、收藏、关注
- notification：消息生成、未读数、聚合逻辑
- moderation：举报、审核、处罚、下架
- admin：后台查询、运营看板、资产审计

## 4. 前端技术栈

### 4.1 主框架

- Next.js 15
- React 19
- TypeScript

### 4.2 选择理由

1. Next.js 适合社区类产品
   帖子详情页、资源页、首页列表都适合服务端渲染或混合渲染，利于 SEO 和首屏性能。

2. React 生态成熟
   组件生态丰富，表单、状态管理、富文本、图片上传、后台表格等常见需求都有成熟解决方案。

3. TypeScript 有利于长期维护
   前后端接口字段多、状态多、权限多，使用 TypeScript 能降低联调和迭代成本。

### 4.3 UI 与工程配套

- Tailwind CSS 4：快速构建页面和后台界面。
- shadcn/ui：用于表单、弹窗、表格、下拉菜单等基础组件。
- Radix UI：提供无障碍和可组合的底层交互能力。
- React Hook Form：管理复杂表单，如发帖、申诉、后台加币。
- Zod：前端表单与接口参数校验。

### 4.4 数据与状态管理

- TanStack Query：负责接口请求、缓存、分页、乐观更新。
- Zustand：管理少量全局状态，如登录态、主题、消息未读数。

### 4.5 富文本与上传建议

- 编辑器：Tiptap
- 图片上传：前端直传对象存储，后端签发上传凭证或走统一上传接口。

### 4.6 后台实现建议

后台不建议单独开第二套前端项目。推荐使用同一个 Next.js 工程，通过 `/admin` 路由分组实现后台页面。

优点：

1. 共享登录逻辑、设计体系和通用组件。
2. 降低前端维护成本。
3. 两人团队更适合单仓统一开发。

## 5. 后端技术栈

### 5.1 主框架

- Java 21
- Spring Boot 3.4
- Spring Web MVC
- Spring Validation
- Spring Security
- MyBatis-Plus
- Flyway
- springdoc-openapi

### 5.2 选择理由

1. Spring Boot 生态成熟
   Java 在企业级后台、权限、交易、审计、调度、运维等场景中生态完整，适合承载这类带资产流转的社区系统。

2. Spring MVC 更适合当前阶段
   当前系统核心压力在数据库事务、权限和后台管理，不在超高并发 IO。相比 WebFlux，Spring MVC 更稳定、心智负担更低，更适合两人团队快速落地。

3. MyBatis-Plus 更适合业务型系统
   社区系统会有大量列表筛选、后台查询、流水审计、交易记录等 SQL 导向场景。MyBatis-Plus 适合快速完成 CRUD，同时保留复杂 SQL 的直接控制力。

4. Flyway 适合持续迭代
   后续积分、支付、标签、推荐等新增能力都需要稳定的数据库版本迁移能力，Flyway 在 Java 项目中足够成熟。

5. Spring Security 更适合权限体系
   用户端、后台端、管理员权限、禁言封禁校验都可以统一落在 Spring Security 体系内，便于后续扩展。

### 5.3 后端工程配套

- Maven：构建、依赖管理、插件统一管理。
- Jackson：JSON 序列化与反序列化。
- HikariCP：数据库连接池。
- MapStruct：DTO、VO、Entity 映射。
- Lombok：减少样板代码。
- jjwt：JWT 生成与校验。
- Spring Boot Actuator：健康检查与运行指标。

## 6. 数据存储与中间件

## 6.1 主数据库：PostgreSQL

推荐版本：PostgreSQL 16

### PostgreSQL 适用原因

1. 事务强一致性好，适合金币、冻结、发放、退款等资产操作。
2. 结构化数据能力强，适合帖子、评论、交易、消息、举报等核心对象。
3. 支持 JSONB，可承载部分扩展字段，降低 V1 初期表结构复杂度。
4. 支持全文搜索，可作为 V1 搜索方案的基础，不必一开始引入 Elasticsearch。

### 建议承载的数据

- 用户与账号状态
- 钱包与金币流水
- 帖子、评论、点赞、收藏、关注
- 资源交易、申诉、退款
- 悬赏帖、答案、采纳、超时退回
- 消息记录、未读状态
- 举报、处罚、后台操作日志

## 6.2 Redis

推荐版本：Redis 7

### 适用场景

1. 缓存热点列表
   如首页帖子列表、热门帖子排行、帖子计数快照。

2. 会话与验证码
   如短信验证码、登录限流、短时登录态辅助缓存。

3. 异步事件与轻量队列
   用作 Redis Stream 的事件通道，承载通知下发、聚合、延迟处理后的消费事件。

4. 短期未读计数缓存
   可作为读性能优化层，但最终消息状态仍以数据库为准。

5. 分布式锁与限流
   通过 Redisson 支持幂等控制、重复提交保护和部分定时任务的集群互斥执行。

### 不建议承担的职责

- 不作为钱包余额的最终数据源。
- 不作为订单状态或交易状态的唯一记录。

## 6.3 对象存储

推荐方案：

- 开发环境：MinIO
- 生产环境：AWS S3、阿里云 OSS、腾讯云 COS 三选一

### 存储内容

- 用户头像
- 帖子图片
- 申诉补充截图
- 后续可能接入的资源附件

### 说明

当前 PRD 中资源帖的隐藏内容可以是链接、提取码、说明文本，因此 V1 不强制做大文件托管系统。对象存储主要承担图片和基础附件能力。

## 7. 异步任务与调度

## 7.1 推荐方案：Redis Stream + Quartz + Redisson

- Redis Stream：承载轻量异步业务事件。
- Quartz：承载定时任务与周期任务。
- Redisson：承载分布式锁、延迟结构和幂等辅助能力。

### 选择理由

1. 相比 Kafka 或 RocketMQ 更轻量
   对两人团队更友好，部署和维护成本更低。

2. 更适合 Java 单体应用
   Spring Boot 接入 Redis 和 Quartz 的成熟度高，不需要引入额外的重型调度或消息平台。

3. 能满足当前任务类型
   包括通知异步写入、消息聚合、悬赏超时扫描、申诉超时提醒、运营日报统计等。

### 适合放到异步任务中的操作

- 生成通知消息
- 更新未读计数
- 聚合点赞通知
- 扫描已过期悬赏帖并执行退回
- 生成运营统计快照

### 不适合放到异步任务中的操作

- 购买资源的核心扣款
- 发布悬赏时的冻结操作
- 采纳答案时的赏金发放

说明：以上核心资产动作必须在主事务中完成，不能异步延迟执行。

### 7.2 调度建议

- Quartz 负责固定时间或周期性任务。
- Redis Stream 负责短链路异步事件消费。
- 如果后期出现更复杂的运营任务编排，再评估接入 XXL-JOB。

## 8. 鉴权与安全方案

## 8.1 登录态方案

推荐采用：

- Access Token：短期有效 JWT
- Refresh Token：较长期有效，支持刷新登录态

### 登录态方案适用原因

1. 前后端分离模式下实现简单。
2. 用户端和后台端可复用同一套认证体系。
3. 后续接入 App 或小程序时仍可沿用。

## 8.2 权限控制

后端必须执行权限校验，前端只做体验层限制。重点校验场景：

- 用户是否登录
- 是否本人帖子
- 是否管理员
- 是否已禁言或封禁
- 是否已购买资源
- 是否有权采纳答案

## 8.3 安全建议

- 密码使用 Spring Security 的 Argon2PasswordEncoder 或 BCryptPasswordEncoder。
- 上传内容需限制文件类型和大小。
- 关键接口增加防重复提交机制，优先使用业务唯一约束 + Redis 短锁双重控制。
- 登录、购买、采纳、加币、扣币等操作记录审计日志。
- 对评论和帖子内容预留敏感词过滤和审核能力。

## 9. 搜索与列表方案

V1 推荐使用 PostgreSQL 全文搜索和普通索引，不引入 Elasticsearch。

### 理由

1. 当前帖子体量初期不大。
2. 只需支持标题和正文关键字检索。
3. 可减少一套独立搜索系统的部署和维护成本。

### 后续扩展条件

当出现以下情况时再考虑 Elasticsearch 或 OpenSearch：

- 帖子规模达到百万级
- 检索排序和召回策略明显复杂化
- 需要高亮、同义词、复杂聚合分析

## 10. 消息与通知设计建议

消息通知采用“数据库存明细 + Redis Stream 做异步分发 + Redis 做未读计数缓存”的方式。

### 建议结构

- 通知明细表：记录每次通知事件
- 用户未读计数：可数据库落表，也可数据库 + Redis 缓存
- 聚合逻辑：由后台消费者异步执行

### 这样设计的原因

1. 数据可靠，便于追溯和补偿。
2. 支持消息已读、全部已读和跳转关联对象。
3. 后续做推送、邮件、短信提醒时可在同一事件模型上扩展。

## 12. 日志、监控与可观测性

## 12.1 日志

推荐：

- Logback + JSON Encoder
- Nginx access log

### 关键日志必须覆盖

- 登录和鉴权失败
- 购买资源
- 悬赏发布
- 采纳答案
- 悬赏退回
- 后台加币、扣币、禁言、封禁
- 举报处理和退款处理

## 12.2 错误追踪

推荐：Sentry

### 选择 Sentry 的原因

1. 配置简单。
2. 前后端都可接入。
3. 两人团队非常需要快速定位线上异常。

## 12.3 指标监控

V1 可分两步走：

1. 起步阶段：依赖应用日志 + Sentry + 基础运营看板
2. 稳定后：接入 Prometheus + Grafana 做接口耗时、错误率、队列积压监控

## 13. 部署方案

## 13.1 环境划分

- 本地开发环境
- 测试环境
- 生产环境

## 13.2 部署建议

推荐使用 Docker Compose 管理以下服务：

- web：Next.js
- api：Spring Boot API
- postgres：主数据库
- redis：缓存和队列
- minio：对象存储（开发或测试环境）
- nginx：统一入口

### 原因

1. 上手快，适合小团队。
2. 环境一致性较好。
3. 后续可平滑迁移到 Kubernetes 或云容器服务。

## 13.3 服务器建议

V1 初期建议：

- 2 核 4G 起步用于测试环境
- 4 核 8G 起步用于正式环境

如果图片和资源附件较少，该配置足以支撑早期社区产品上线。

## 14. CI/CD 建议

推荐：GitHub Actions

### 基础流程

1. 代码提交后执行前后端 lint。
2. 执行后端单元测试和接口测试。
3. 执行前端构建和基础测试。
4. 构建 Docker 镜像。
5. 推送到测试环境或生产环境。

## 15. 推荐目录结构

建议采用单仓结构：

```text
lenjoy-bbs/
  apps/
    web/
    api/
  packages/
    shared-types/
  infra/
    docker/
    nginx/
  docs/
    需求文档.md
    用户故事.md
    技术栈方案.md
```

说明：

1. `apps/web` 放前端站点和后台页面。
2. `apps/api` 放 Spring Boot 服务。
3. `infra` 放部署配置。
4. `docs` 统一管理产品和技术文档。

## 16. 不推荐的方案

当前阶段不建议使用以下方案：

1. 微服务架构
   当前没有组织规模和业务复杂度支撑，拆分后会显著增加联调和运维成本。

2. Elasticsearch 作为 V1 搜索基础设施
   当前搜索需求并不复杂，PostgreSQL 足够。

3. Kafka 或 RocketMQ
   当前异步任务量不大，Redis 队列更合适。

4. 单独再做一套后台前端
   会让前端成本和组件维护成本翻倍。

5. 过早引入复杂风控系统
   V1 先把审计日志、操作日志和基础限制做好，再根据业务情况逐步升级。

## 18. 一句话结论

如果目标是让两人团队尽快把社区产品 V1 做出来，同时保证金币和交易闭环的稳定性，最合适的方案是：

使用 `Next.js + TypeScript` 作为前端，`Spring Boot + PostgreSQL + Redis Stream/Redisson + Quartz` 作为后端与异步任务基础设施，采用模块化单体架构，通过 Docker Compose 完成部署。
